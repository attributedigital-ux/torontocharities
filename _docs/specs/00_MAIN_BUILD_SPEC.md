## 4. Auth model

| Role | Can do |
|---|---|
| Public (no auth) | View profiles, events, categories, guides. Submit a charity. Start a claim. |
| `charity_owner` (after claim verification) | View their own charity dashboard. Edit description, contact details, image. Add events manually for their charity. Toggle event-source opt-out. |
| `admin` (manual promotion) | All charity_owner abilities for any charity. Approve/reject events. Manage event sources. Verify/deny claims. Edit guides. |

**Claim verification mechanism** (from event spec references to `linkback_verified_at`):
1. User submits claim form for charity X with their email + URL on charity X's site where they'll add the linkback
2. System generates `claim_token`, emails user a magic-link to confirm email + a unique HTML snippet to add to their site
3. User adds snippet to their site, hits "verify" in the dashboard
4. Cron job (`verify-linkbacks.js` per event spec line 528) crawls the URL, confirms snippet present, sets `linkback_verified_at`
5. Verified charities: events auto-approve from their feeds; profile shows "verified" badge

## 6. Cost-optimization plan for Claude pipelines

This is the answer to "make the agents economical." Two pipelines call Claude — both apply the same cost levers.

### Model choice

**Use Claude Haiku 4.5** (`claude-haiku-4-5-20251001`) for both pipelines. The CRA-description task ("write 80-150 words from a website excerpt, no marketing language") and the event-enrichment task ("clean a title, rewrite a description, extract dates and cost, categorize, score confidence") are both well within Haiku's capability envelope. The sub-specs default to Sonnet but the work doesn't need it.

Fallback to Sonnet 4.6 only if a quality-review pass flags Haiku outputs as failing the standard.

### Anthropic Message Batches API

The CRA description run is a one-shot bulk job over 2,500-3,500 charities — exactly the Batches API use case. **Use it.** 50% discount on every input and output token. 24-hour SLA but the job has no urgency.

Event enrichment is recurring small batches (50/week) — too small for Batches API to help. Run inline.

### Prompt caching

Both prompts have a substantial static portion (style rules, format spec, banned words, schema for the JSON response). **Cache that portion** using `cache_control: { type: 'ephemeral' }` on the system message. The variable portion (per-charity or per-event data) is small.

Expected hit rate: very high during the CRA one-shot (every call within a 5-min burst). High during event enrichment (cron runs daily, so cache is cold each day — but within a single cron run with N events, the cache is warm).

### Strict `max_tokens` budgets

- CRA description: `max_tokens: 600` (cap on 80-150 word output is ~300 tokens; 600 covers retries and longer outputs)
- Event enrichment: `max_tokens: 800` (JSON object with multiple fields)

### Estimated cost

| Job | Volume | Model | Batch | Cached | Est. cost |
|---|---|---|---|---|---|
| CRA descriptions (one-shot) | 2,500 calls | Haiku 4.5 | yes | yes | **~$8** |
| Event enrichment (weekly) | 50 calls/week | Haiku 4.5 | no | yes | **~$0.50/week** = $26/year |

For comparison, the same workload on Sonnet 4.6 without batch or caching: ~$40 one-shot + ~$2/week. Worth optimizing.

### What I will NOT do

- No Opus calls in either pipeline. Both tasks are pattern work, not reasoning work.
- No retry-without-backoff loops that could pile up costs on transient errors.
- No "verify the output with a second model call" patterns. Trust the model output, sample-review human-side instead.
